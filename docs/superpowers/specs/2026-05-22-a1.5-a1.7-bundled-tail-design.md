# A1.5 + A1.7 + Phase 2.B — Bundled-stack tail — Design

**Date:** 2026-05-22
**Status:** Shipped (v0.8.0 A1.5 + A1.7). Historical design reference.
**Owner:** samkintop

## Goal

Finish the bundled-system work: harden the docker stack against supply-chain + lateral-movement risk (A1.5), give the operator a single pre-flight + first-launch acknowledgement gate (A1.7), and surface embed/rerank status in the Models panel so the bundled stack is fully observable from one screen (Phase 2.B).

Three sections that interact only through the docker-compose file. Implementation can be sequenced or done as one branch.

## Non-goals (v1)

- **C-track work**: column encryption (C6), audit logging (C4), de-id pipeline (C5), right-to-erasure (C9). Those are their own phases with non-trivial scope. A1.5 only absorbs C2 (docker hardening) as the roadmap states.
- **HLH_MASTER_KEY check in `make doctor`**: feature doesn't exist yet (lives in C6). Skip the check until C6 lands.
- **Authelia / reverse-proxy reachability check**: Sam-specific infra. The pre-flight will skip if not configured rather than fail.
- **Rewriting `model_puller.pull_model` to handle multi-file HF repos**: the original spec §10 follow-up. Phase 2.B takes the lighter path — surface infinity's own state via a small read-only endpoint, not download embed/rerank ourselves.
- **No-0.0.0.0 binding rule from the original roadmap**: explicitly reversed. The bundled stack is user-agnostic and binds 0.0.0.0 on the services that need host access (api, ui, search). Internal-only services (chat, infer) stay unexposed.

### MODEL_REGISTRY pinning

Today's `MODEL_REGISTRY` entries carry no `revision` and no `sha256`. The puller already has the sha256 mismatch path coded but it's never exercised because the spec is empty.

For each currently-shipped chat tier, add:
- `revision: "main"` (until we have a reason to pin a specific HF revision; just having the field documented is the win — operators can override)
- `sha256: "<computed>"` (compute after a clean pull on Sam's host, paste back)
- `expected_bytes: <int>` (already optional; populate for at least cpu-min so the disk pre-flight has a number to compare against)

Process (implementation step):
1. Pull each tier's model once on a clean host.
2. Compute `sha256sum` of the resulting file.
3. Note `wc -c` for `expected_bytes`.
4. Paste both into `MODEL_REGISTRY`.
5. Re-run pull on a different fresh volume; verify the integrity check passes (happy path).
6. **Also verify the corruption path** — see verify script §5.1's "corrupt a partial file mid-pull, re-pull, assert sha256 check rejects". The pinning is only useful if the rejection actually fires; the verify script exercises that.

If unsloth bumps a quant file (rare but possible — uploads can be re-quantized), the pinned sha will fail and the operator gets a clear "sha256 mismatch" error. Operator decides whether to update the pin or pull a different file. **Documented as expected behavior.**

### Puller — disk pre-flight

Before opening the stream:

```python
async def _check_disk_space(dest_dir: Path, expected_bytes: int | None) -> None:
    """Refuse pull if free space - expected_bytes < 5 GB."""
    import shutil
    if expected_bytes is None:
        return  # can't check without a target; allow (logs a warning)
    free = shutil.disk_usage(dest_dir).free
    needed = expected_bytes + (5 * 1024 ** 3)  # 5 GB headroom
    if free < needed:
        raise InsufficientDiskError(
            f"need {needed:,} bytes ({needed / 1024**3:.1f} GB) — only {free:,} free"
        )
```

Call site in `pull_model` before the dest mkdir. On `InsufficientDiskError`, mark the row `status='failed', error_message='insufficient disk: <details>'` and return (matches the existing failure path shape).

### Container hardening

Apply to `hlh_chat`, `hlh_infer`, `hlh_api`, `hlh_ui`, `hlh_search`. Same template per service:

```yaml
  hlh_chat:
    # ...existing image, command, env, volumes...
    user: "1000:1000"        # non-root inside the container
    read_only: true          # immutable rootfs
    tmpfs:
      - /tmp
    cap_drop:
      - ALL
    security_opt:
      - no-new-privileges:true
    mem_limit: 3g            # per-tier override via .env (see below)
```

Per-tier mem_limit for `hlh_chat` (driven by `HLH_CHAT_MEM`):
- cpu-min: 3g (Qwen 0.8B is ~1.5 GB peak; 3g leaves headroom)
- cpu-std: 6g (MedGemma 4B Q4_K_M)
- gpu-8gb: 8g
- gpu-16gb: 18g
- gpu-24gb+: 22g

`hlh_infer` mem_limit: 4g (bge-m3 + bge-reranker-v2-m3 both ~1 GB at Q4-equivalent; runtime peak ~3 GB).

`hlh_api` / `hlh_ui` / `hlh_search`: no mem_limit (typical operator hosts can't predict; leave to system OOM).

**Caveat about read_only + tmpfs**: some services write to /tmp or other paths during boot. Test each — if a service can't boot read_only, add the specific writable path to `tmpfs` rather than disabling read_only. Documented per-service in implementation.

### Network split

Today: every service on `hlh_default`. The chat sidecar shouldn't be reachable from outside the api container.

New:

```yaml
networks:
  hlh_default:        # existing; api + ui + search join
    driver: bridge
  hlh_inference:      # new; api + chat + infer join
    driver: bridge
    internal: true    # no egress except through api
```

Per service:

| Service | hlh_default | hlh_inference |
|---|---|---|
| hlh_db | ✓ | — |
| hlh_api | ✓ | ✓ |
| hlh_ui | ✓ | — |
| hlh_search | ✓ | — (needs egress for upstream engines) |
| hlh_chat | — | ✓ |
| hlh_infer | ✓ | ✓ (egress needed for HF; defense-in-depth via container hardening) |

`internal: true` means containers on that network can't reach the internet. Chat reads local model files only; chat moves to `hlh_inference`-only. **Infer joins both networks** because infinity downloads model weights from HuggingFace on its own — that download path needs egress. After the cache fills it could in principle drop `hlh_default`, but doing so requires manual operator intervention; instead we accept the egress and rely on container hardening (read_only, cap_drop, no host ports) to bound the blast radius.

### Bind addresses

The roadmap originally said "no service binds 0.0.0.0 — all binds to 100.114.205.53." That was Sam-specific (his host has Tailscale at that IP). **Reversed:**

- `hlh_api`: binds `0.0.0.0:9600` (host-facing)
- `hlh_ui`: binds `0.0.0.0:9604`
- `hlh_search`: binds `0.0.0.0:9612`
- `hlh_chat`, `hlh_infer`, `hlh_db`: no host ports (internal only)

Confirmed via `docker ps`. No change needed; flagged here so it doesn't get "fixed" later by someone reading the old roadmap text.

## §2 — A1.7: Pre-flight + first-launch ack

### `make doctor` (or `python -m hlh.doctor`)

Single command. Runs on container start (logged) and on operator demand via `docker exec hlh_api python -m hlh.doctor`.

Checks (each returns green / yellow / red + a remediation line):

| Check | Green | Yellow | Red |
|---|---|---|---|
| DB pool healthy | pool connects | — | pool fails to acquire |
| Schema applied | `system_profile.id=1` row exists | — | row missing |
| `setup_complete` | true | false (just first-time setup) | — |
| Chat sidecar reachable | `GET hlh_chat:9610/health` 200 | unhealthy (still booting) | unreachable |
| Infer sidecar reachable | `GET hlh_infer:9611/health` 200 | unhealthy (still pulling models) | unreachable |
| Search sidecar reachable | `GET hlh_search:8080/healthz` 200 | unhealthy | unreachable |
| Safeguard prompt version present | `from services.safeguards import SAFEGUARD_VERSION` succeeds and the value is non-empty | — | import fails or value blank (means B0 was reverted / module deleted) |
| Disk free under `/data` | ≥10 GB | 5–10 GB | <5 GB |
| Disk free under `/models` | ≥10 GB | 5–10 GB | <5 GB |
| `PROVIDER_KEY_ENCRYPTION_KEY` set | reuse `services.crypto._key()` returns 32 bytes successfully | env unset (cleartext fallback active) | env set but `services.crypto._key()` raises (malformed base64 OR wrong byte length) |
| HF token configured | yes (via DB or env) | unset (gated models will fail) | — |

**Safeguard version scope clarification.** No singleton "current safeguard version" exists in DB today — `messages.safeguard_version` is per-row (stamped on every assistant write). Comparing the code's `SAFEGUARD_VERSION` to a DB singleton would require adding a `safeguards_state` table. Not worth it for v1; **the check is scoped to "code module imports + value present"**. If we add a singleton in a future safeguards phase, upgrade the check to compare.

**PROVIDER_KEY validation logic.** Don't re-implement the check inline. Call into `services.crypto._key()` — it already validates: returns `None` for unset (→ yellow), returns 32 bytes for valid (→ green), raises `RuntimeError` for malformed base64 or wrong-length (→ red, surface the message in the remediation line).

Skipped checks (not in scope today):
- LUKS (host-distro-specific; defer to C1)
- Backrest snapshot freshness (defer to C1)
- HLH_MASTER_KEY (defer to C6)
- Authelia reachability (Sam-specific)

Output format (text, no PHI):

```
$ python -m hlh.doctor
✓ DB pool healthy
✓ Schema applied
⚠ setup_complete=false — visit Settings → System to pick a tier
✓ Chat sidecar reachable
⚠ Infer sidecar unhealthy — first-boot HF download in progress (this is normal for ~5-15 min)
✓ Search sidecar reachable
✓ Safeguard prompt version: b0-2026-05-22b (matches code)
✓ Disk free under /data: 142 GB
✓ Disk free under /models: 142 GB
⚠ PROVIDER_KEY_ENCRYPTION_KEY unset — provider api_keys + HF token stored in cleartext
⚠ HF token unset — MedGemma pulls will fail with 401

3 warnings, 0 errors. Safe to proceed.
```

Exit code: 0 if no reds, 1 if any red. CI-friendly.

### Surface in UI

New endpoint `GET /api/system/doctor` returns the same checks as JSON:

```json
{
  "checks": [
    {"name": "db_pool", "status": "ok", "detail": "healthy"},
    {"name": "setup_complete", "status": "warn", "detail": "visit Settings → System"},
    ...
  ],
  "summary": {"ok": 8, "warn": 3, "error": 0}
}
```

System tab gets a "Pre-flight" expandable section that hits this endpoint on mount and refresh, renders the same table with colored badges. Refresh button.

### First-launch acknowledgement modal

Schema addition (idempotent):
```sql
ALTER TABLE system_profile ADD COLUMN IF NOT EXISTS acknowledged_at TIMESTAMPTZ;
```

When `setup_complete=true AND acknowledged_at IS NULL`, the System tab shows a modal:

> **HomeLab Health is a personal-records tool, not a medical service.**
>
> What it is:
> - A place to keep your records and notes
> - A chat that can explain symptoms, conditions, and lab terms in plain language
> - *(if `hlh_search` is healthy)* Optional web search to ground answers in current information
>
> What it isn't:
> - A doctor. It doesn't diagnose. It will refuse.
> - A pharmacist. It doesn't opine on drug interactions or dosages. It will refuse.
> - A 911 alternative. If you're in crisis, it will show you the hotline.
>
> By continuing, you acknowledge this is educational, not medical advice.
>
> [ ] I understand
> [Continue]

The "Optional web search" bullet renders only when `hlh_search` reports healthy via `GET /api/system/doctor` (the same endpoint backing the Pre-flight section). If the search sidecar isn't running (operator dropped it from the `bundled` profile, or it's still booting), the bullet stays hidden — no broken promises in the friend's first impression.

Checkbox is required to enable the button. On click: `PATCH /api/system/profile` with `acknowledged_at = NOW()`. Modal closes.

Cannot be re-shown via UI. To reset: `UPDATE system_profile SET acknowledged_at = NULL WHERE id = 1;` (operator-only).

This is the B2 first-launch ack landed early as part of A1.7 (the roadmap allows this — B2 explicitly says the first-launch modal folds into A1.7).

## §3 — Phase 2.B: Embed + rerank visibility in Models panel

The Phase 2.A follow-up: today the Models panel only shows chat rows because `MODEL_REGISTRY['embed']` / `['rerank']` are still `None` (infinity downloads them itself).

**Lighter approach (chosen): synthesize rows from the bundled provider list.**

Frontend change only. The Models panel queries `/api/providers` (already available), filters to `is_bundled=TRUE`, and renders synthetic rows for embed + rerank alongside the existing puller-driven chat rows:

| Role | Model | Status | Progress | License | Action |
|---|---|---|---|---|---|
| chat | (puller-managed) | pending/pulling/ready | progress bar | apache-2.0 / gemma | Pull |
| embed | `BAAI/bge-m3` | sidecar-managed | (none) | mit | Test |
| rerank | `BAAI/bge-reranker-v2-m3` | sidecar-managed | (none) | apache-2.0 | Test |

**Sidecar-managed status:**
- Synthesized client-side from the bundled provider's `last_verified_status`. Column confirmed to exist on `providers` (`schema.sql:312`); populated by every `/test` probe.
- Mapping: `last_verified_status = 'ok'` → "ready". `null` (no probe yet) → "loading". Any value starting with `'error:'` → "error" with the message surfaced.
- Test button on these rows fires `POST /api/providers/{id}/test` (already implemented in T6). The role-aware probe verifies the sidecar can serve. The first auto-probe runs on Models panel mount, so `last_verified_status` becomes non-null within ~1s of the panel opening (not 30s as initially assumed).

**Progress for first-boot HF download:** infinity exposes no progress endpoint we can scrape easily. Acceptable: show "loading model weights…" with a spinner; no percent. Polling rules:
- Models panel polls the test endpoint every 5s **while status is "loading"**.
- **Hard cap: 60 attempts (5 min total).** After 60 failed probes, flip the row to "error: sidecar didn't come up within 5 min — check `docker logs hlh_infer`".
- Polling stops on first "ready" or "error" — only "loading" keeps it going.
- Manual refresh (page reload or Test button click) resets the attempt counter so the operator isn't stuck if the first 5 min wasn't enough.

**No change to model_puller.py.** The puller stays chat-only. Embed/rerank are operationally separate (sidecar-managed).

### UI text

Add a small note at the top of the Models panel under the existing helper text:

> Embed and rerank weights are downloaded automatically by `hlh_infer` on first boot. They appear as "loading" until the sidecar reports healthy, then "ready". No pull button — the sidecar manages itself.

## §6 — Known limitations + follow-ups

- **Image pin staleness over time**: pinned tags get old. Add `make security` (C8) eventually to scan for new versions. Not in this phase.
- **MedGemma sha pinning blocks unsloth re-quantizations**: if unsloth re-uploads a quant file (rare), the integrity check fails. Documented behavior; operator updates the pin manually.
- **`hlh_infer` egress on first boot**: stays on `hlh_default` for HF access. After first cache, operator can manually flip to internal-only — not automated. Future follow-up: detect cache-present state on boot and auto-flip.
- **Doctor's PROVIDER_KEY_ENCRYPTION_KEY check is warn-only**: cleartext-fallback is currently expected. Becomes red when C6 ships and the key is mandatory.
- **Doctor doesn't check LUKS / backrest / Authelia / master key**: deferred to C1 + C6. The shape is extensible — adding a check is a 5-line PR.
- **Phase 2.B doesn't show real pull progress for embed/rerank**: infinity has no easy progress endpoint. Status is binary (loading / ready). Acceptable for v1.
- **First-launch modal cannot be re-shown via UI**: operator-only reset via SQL. Could become a Settings → System "Reset acknowledgement" button later.
