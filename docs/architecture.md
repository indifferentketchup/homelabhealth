# homelabhealth — Architecture

Live architecture for `main` as of 2026-06-12, after the 2026-06-09
`fork-lift-wave-1` commits. For release history, see [CHANGELOG.md](../CHANGELOG.md).

## Overview

homelabhealth is a single-user, self-hosted RAG chat application for personal
health records. The current stack is a React/Vite frontend, a FastAPI backend,
PostgreSQL with pgvector, a bundled llama.cpp router service for chat plus
retrieval models, and a bundled SearXNG service for web search.

The important current-state correction is this: the bundled stack no longer
uses a separate `hlh_infer` embeddings/rerank sidecar in the live compose file.
Bundled chat, embeddings, rerank, and vision all resolve to `hlh_chat`.

## Runtime services

| Service | Role | Notes |
|---------|------|-------|
| `hlh_ui` | Static frontend + `/api` proxy | Nginx container on host port `9604` |
| `hlh_api` | FastAPI application | All business logic, auth, RAG, orchestration |
| `hlh_db` | PostgreSQL 16 + pgvector | Persistent app data |
| `hlh_chat_cpu` / `hlh_chat_gpu` | Bundled llama.cpp router | Internal-only on `hlh_inference`; serves chat/embed/rerank/vision presets |
| `hlh_search` | Bundled SearXNG | Host port `9612`, internal app dependency for web search |

Not a compose service:

- `hlh_orchestra` is now a bootstrap-only image used by `install.sh` / raw
  `docker run` startup. It is not part of the steady-state compose topology.

## Networks and volumes

- `hlh_default`: UI, API, DB, Search
- `hlh_inference`: internal-only network for API ↔ chat router traffic

Persistent volumes:

- `hlh_db_data`: Postgres data
- `hlh_models`: GGUF weights, router assets, active model symlinks
- `hlh_uploads`: uploaded source files
- `hlh_keys`: generated encryption keys
- `hlh_branding`: user/profile branding assets
- `hlh_history`: exported history files

## Request flow

### Standard chat flow

1. Browser hits `hlh_ui`, which proxies `/api/*` to `hlh_api`.
2. `hlh_api` authenticates via `hlh_session` cookie.
3. `backend/routers/chats.py` assembles system context, sources, safeguards,
   approvals, and provider/model resolution.
4. Retrieval uses `backend/services/rag.py`.
5. Bundled inference resolves through `backend/services/bundled_providers.py`
   and `backend/services/provider_client.py` to `http://hlh_chat:9610`.
6. Streaming is exposed back to the UI through the durable/SSE orchestration
   path consumed by `frontend/src/hooks/useStreamOrchestrator.js`.

### Source ingest flow

1. Upload lands in `backend/routers/sources.py`.
2. `backend/services/vision.py` handles PDFs/images when needed.
3. `backend/services/chunking.py` chunks parsed text.
4. `backend/services/deid.py` redacts when enabled.
5. Embeddings are generated by the active provider. For bundled mode, this is
   the `hlh_chat` router preset for `qwen3-embed` (Qwen3-Embedding-0.6B).
6. Chunks are stored in `source_chunks` with `vector(1024)` embeddings.

## Bundled provider resolution

The current bundled provider rows are all seeded by
`backend/services/bundled_providers.py`:

| Name | URL | Role |
|------|-----|------|
| HomeLab Health AI · Chat | `http://hlh_chat:9610` | chat |
| HomeLab Health AI · Embed | `http://hlh_chat:9610` | embed |
| HomeLab Health AI · Rerank | `http://hlh_chat:9610` | rerank |

Tier changes and app startup both call `apply_bundled_bindings(...)`, which
rewrites the global embed/rerank bindings and bundled chat defaults.

## Backend surfaces

Key routers under `backend/routers/`:

- `auth.py`, `profile.py`
- `workspaces.py`, `workspace_memory.py`, `workspace_context_files.py`
- `chats.py`, `chats_crud.py`, `sources.py`, `notes.py`, `history.py`
- `demo.py`, `analytics.py`, `eval.py` (mounted in `main.py`; exposes
  `/api/eval/groundedness|helpfulness|retrieval-relevance`)
- `models.py`, `inference.py`, `providers.py`
- `settings.py`, `system.py`, `search.py`, `searxng.py`
- `audit.py`, `custom_instructions.py`, `memory.py`

Key service clusters:

- `services/inference_job.py`: request execution, approvals, streaming writes
- `services/process_pool.py`: process-pool orchestration added in Wave 4
- `services/conductor.py`: multi-perspective wave scheduler
- `services/approval_gate.py`: pause/resume human approval flow
- `services/memory/`: 3-tier SQLite memory engine (agent-search index; see its
  module docstring for the full memory-store ownership map)
- `services/memory_tools.py`: LangMem-style memory extraction/search tools
- `services/patient_profile.py`: structured longitudinal patient profile
  (`workspace_patient_profile` JSONB table), injected at inference and
  conflict-resolved on update; the inference-time authoritative store for
  durable health facts
- `services/safeguards_engine.py`: fixed-rule safeguard engine — keyword-matches the
  user query against five rules (diagnosis, emergency, self-harm, medication,
  medication-combination) and resolves overlaps to the highest-priority match;
  `safeguards.py` turns the result into contextual directives or the self-harm full
  prompt. Trimmed 1129->257 lines from a generic guideline framework 2026-06-15
  (openspec `trim-safeguards-engine`)
- `services/prompt_assembly.py`: `_assembled_system_prompt` + streaming/title
  helpers (promoted out of `routers/chats.py` to break a service->router cycle;
  shared by the SSE and durable paths)
- `services/provider_client.py`: provider resolution + `async_llm_call`, the
  single non-streaming completion helper used by all background LLM callers
- `services/summarization.py`: shared summary prompt + medical-fact pinning,
  used by both `compaction.py` and `pruning.py`
- `services/deep_research.py`: iterative SearXNG research loop (query -> search
  -> summarize -> reflect -> synthesize with citations); de-identified on the
  external tier
- `services/eval_judge.py`: LLM-as-judge + the async-background groundedness
  scorer fired after a response (never inline on bundled inference)
- `services/stall_detector.py`, `services/context_handoff.py`: pure-function
  helpers for orchestration loop-detection and wave-output compression

New tables/columns this wave: `workspace_patient_profile` (JSONB); on `messages`:
`groundedness_score`, `retry_count`, `max_retries`, `orchestration_cursor`.

## Lifespan / startup

`backend/main.py` currently does:

1. ensure encryption/orchestra tokens
2. install log redaction and deprecated-env warnings
3. init DB pool and apply schema
4. ensure first admin user exists
5. migrate legacy chat paths
6. seed bundled model registry and reset orphaned pulls
7. apply bundled provider bindings for the active tier
8. log startup banner
9. start the stale-streaming sweeper task (liveness-aware, with a
   `retry_count`/`max_retries` budget before promoting a row to `failed`)
